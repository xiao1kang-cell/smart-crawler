# Anker VOC 新计费方案 v2

日期：2026-06-25

## 1. 方案目标

本方案用于替代原先仅按“登录页面 / 非登录页面”两档计费的月结方式。新方案同时区分：

- 优先级：由 Anker 在提交任务时指定，支持 `P0` / `P1` / `explore`；`P2` 当前不启用，如接口传入则按 `P1` 计费
- 是否登录：`login_required=true / false`
- 任务类型：评论采集、商品详情、评论存在检查、星级统计、探索采集等
- 资源占用：账号池、浏览器、代理、服务器、监控告警、回调推送

推荐原则：

1. `P1` 作为常规生产任务，价格沿用旧结算口径，便于平滑切换。
2. `P0` 单独加价，因为需要抢占调度、预留并发、账号池优先使用和更短 SLA。
3. 当前 Anker 不设 `P2` 折扣档，常规任务统一按 `P1` 价格结算。
4. `explore` 只建议用于非登录态探索，默认返回亚马逊页面公开展示的 Top Reviews，低价但不承诺完整性。
5. 固定成本和流量费用拆开结算，避免把服务器、账号池维护、定制开发混进页面单价里导致后续难对账。

## 2. 计费单位定义

### 2.1 主计费单位：成功页面量

延续当前结算习惯，主计费单位仍为“成功页面量”，按 1000 次为一个价格单位。

一个可计费页面指：

- 任务实际执行；
- 成功返回可用页面或结构化结果；
- 写入统计接口并可按 `task_id` / `biz_source` / `priority` / `login_required` 追溯；
- 不是由系统内部重试导致的重复请求。

内部重试、验证码失败重试、账号切换重试不重复计费，只按最终成功产出的页面计一次。

### 2.2 辅助统计单位

月结单建议同时展示以下统计，但默认不单独收费，除非双方后续约定：

| 指标 | 用途 | 默认是否计费 |
|---|---|---|
| `tasks_submitted` | Anker 提交任务数 | 否 |
| `tasks_success` | 成功任务数 | 否 |
| `pages_fetched` | 成功页面量，主计费口径 | 是 |
| `records_returned` | 返回评论/商品记录数 | 否，作为验收与质量指标 |
| `api_calls` | 内部 API/HTTP 成功请求数 | 否，作为成本核算 |
| `browser_opens` | 浏览器渲染次数 | 否，作为成本核算 |
| `callback_success` | 回调成功数 | 否 |
| `callback_failed` | 回调失败数 | 否，作为 SLA/排障指标 |

## 3. 优先级与 SLA 定义

| 优先级 | 适用场景 | 调度策略 | 建议 SLA | 价格逻辑 |
|---|---|---|---|---|
| `P0` | 舆情危机、差评核实、下架/投诉紧急排查 | 抢占式调度，优先占用账号池和并发 | 4 小时内返回结果或失败原因 | 高价，覆盖预留资源 |
| `P1` | 日常生产、业务方常规同步 | 标准生产队列 | 24 小时内完成 | 沿用旧单价 |
| `explore` | 非登录态探索、竞品快速探测、Top Reviews 默认 8 条 | 尽力而为，不占登录账号池 | 无强 SLA，建议 7 天内完成 | 低价，只适合非登录态 |
| `P2` | 当前不启用；历史或异常传参 | 自动归入 P1 队列 | 按 P1 SLA | 按 P1 价格 |

建议限制：

- 任务优先级由 Anker 控制，系统按 Anker 传入的 `priority` 进行调度、统计和结算。
- `P0` 需要设置月度上限或预留包，否则会挤占常规任务。
- `explore` 默认 `login_required=false`，返回公开页面默认展示内容，不承诺全量评论、不承诺翻页完整性。
- 若 Anker 强制要求 `explore + login_required=true`，应自动转为 `P1` 计费。

## 4. 推荐价格表

单位：人民币元 / 1000 成功页面。

| 优先级 | 登录态页面 `login_required=true` | 非登录态页面 `login_required=false` | 说明 |
|---|---:|---:|---|
| `P0` | 5.30 | 3.60 | 抢占调度、短 SLA、优先账号池 |
| `P1` | 3.30 | 2.20 | 标准生产任务，沿用旧结算基准 |
| `P2` | 3.30 | 2.20 | 当前不启用；如传入则按 P1 归档 |
| `explore` | 不建议；按 P1 归档 | 1.20 | 非登录态探索，默认 Top Reviews 8 条 |

价格公式：

```text
采集流量费 =
  sum(成功页面量 / 1000 * 对应优先级与登录态单价)
```

保留旧口径的兼容关系：

- 登录态 P1 = 3.30 元 / 1000 页面；
- 非登录态 P1 = 2.20 元 / 1000 页面；
- 如果全部任务都归为 P1，则结算金额与旧方案一致。

## 5. 任务类型计费口径

| 任务类型 | 示例接口 / Job | 默认登录态 | 默认优先级 | 计费方式 |
|---|---|---|---|---|
| 评论全量/增量采集 | `AmazonReviewJob` / `accept_single_task` | 登录态 | P1 | 按成功页面量计费 |
| 商品详情采集 | `accept_asin_detail_task` | 非登录态为主 | P1 | 按成功页面量计费 |
| Listing / Search / Top100 | `AmazonListingJob` / `AmazonSearchrankJob` / `AmazonTop100Job` | 非登录态 | P1 | 按成功页面量计费 |
| Rufus / 特殊页面 | `Amazon Rufus` | 登录态 | P1 | 按登录态页面计费 |
| 评论存在检查 | review_id exists check | 非登录态优先 | P0/P1 | 建议按 0.5 个页面折算；P0 仍按 P0 单价 |
| 星级分布统计 | rating distribution | 非登录态优先 | P1 | 建议按 1 个页面折算 |
| explore Top Reviews | 非登录态 Top 8 | 非登录态 | explore | 按 explore 非登录态单价 |

折算建议：

```text
评论存在检查计费页面 = 成功检查数 * 0.5
星级分布统计计费页面 = 成功 ASIN 数 * 1
Top Reviews explore = 成功 ASIN 数 * 1
```

如果某个任务实际需要翻页、登录或浏览器渲染，则按实际 `login_required` 和成功页面量计费。

## 6. 固定费用与定制成本

采集流量费之外，固定费用建议继续独立月结。

### 6.1 服务器与基础设施

沿用原结算方式，按实际配置计费：

| 项目 | 建议口径 |
|---|---|
| 采集服务器 | 按服务器数量 * 单价/月 |
| 代理/静态 IP/住宅 IP | 如 Anker 任务单独使用，按实际成本 + 管理费列示 |
| 存储与日志 | 常规日志保留包含在服务费内；超长期保留另议 |

### 6.2 账号池与运维服务费

新需求新增账号池隔离，建议单独列出：

| 项目 | 建议计费方式 | 说明 |
|---|---|---|
| 本品账号池维护 | 月固定服务费 | 账号健康检查、Cookie 更新、异常账号下线 |
| 竞品账号池维护 | 月固定服务费 | 与本品池隔离，避免相互影响 |
| P0 预留并发 | 月固定预留费或 P0 最低消费 | 保证紧急任务不被普通队列挤占 |
| 监控告警 | 月固定服务费 | 成功率、时效、账号池健康度、积压告警 |

推荐商务条款：

- 如果 Anker 每月 `P0` 用量低但要求 4 小时 SLA，应收取 P0 预留费；
- 如果 Anker 不购买 P0 预留费，则 `P0` 单价可保留，但 SLA 改为“优先处理，不承诺资源预留”；
- 账号由 Anker 提供时，只收维护费；账号由我方采购时，账号成本实报实销另加管理费。

### 6.3 一次性开发费用

以下属于新需求适配开发，不建议摊入页面单价：

| 模块 | 说明 |
|---|---|
| 优先级调度 | `P0/P1/explore` 队列、抢占调度、限流策略；P2 传参按 P1 兼容 |
| 业务方对账字段 | `biz_source`、优先级、登录态、账号池维度入库 |
| 回调推送 | callback、重试、推送记录、失败可查 |
| 统计接口 | 成功率、时效、返回量、用量聚合 |
| 新增检查接口 | 星级分布、指定评论存在检查 |
| 字段补全 | reviewer profile、图片/视频数量、raw_json 等 |

建议作为“一次性项目开发费”或“月度定制开发费”独立报价。

## 7. 成功、失败、缓存复用的计费规则

| 情况 | 是否计费 | 规则 |
|---|---|---|
| 成功返回结果 | 是 | 按成功页面量计费 |
| 系统内部重试后成功 | 是 | 只计最终成功页面，不重复计重试 |
| 账号失效、验证码、风控导致最终失败 | 否 | 记录失败原因，进入 SLA 统计 |
| ASIN 不存在 | 否，或按 0.2 页面收取校验费 | 建议先不计费，降低争议 |
| Anker 参数错误 | 否，或按无效请求统计 | 返回明确错误码，不进入页面计费 |
| 命中缓存复用 | 建议按 20% 页面折算 | 因仍有存储、查询、回调成本，但不消耗采集资源 |
| Anker 重复提交同参任务 | 命中缓存按 20% 折算 | 防止重复任务造成对账争议 |
| 回调失败但结果可查询 | 采集费正常计 | 回调失败进入推送失败统计，不重复采集 |

缓存复用折算公式：

```text
缓存复用费用 = 复用页面量 / 1000 * 对应单价 * 20%
```

## 8. 月结单推荐结构

新结算单建议拆成 5 部分：

1. 采集任务明细：按业务线、任务类型、优先级、登录态展示完成量。
2. 采集流量费：按 `priority + login_required` 聚合计费。
3. 固定成本：服务器、代理、账号池、监控告警。
4. 定制开发/维护：本月新需求开发工时或包月服务。
5. SLA 与质量统计：成功率、P95 时效、失败原因 Top、回调成功率、账号池健康度。

建议最小对账维度：

| 字段 | 说明 |
|---|---|
| `month` | 结算月份 |
| `biz_source` | CD / QMS / ADT / CEMO 等 |
| `task_type` | review / product / listing / exists_check / rating_stats |
| `priority` | P0 / P1 / explore；P2 兼容归入 P1 |
| `login_required` | true / false |
| `success_pages` | 成功页面量 |
| `reused_pages` | 缓存复用页面量 |
| `records_returned` | 返回记录数 |
| `unit_price_per_1k` | 单价 |
| `amount` | 金额 |
| `success_rate` | 成功率 |
| `p95_duration_minutes` | P95 完成时长 |

## 9. 按当前 Anker 量级的示例测算

当前截图中 Anker 业务线完成量：

| 登录态 | 页面量 |
|---|---:|
| 登录页面 | 6,794,428 |
| 非登录页面 | 10,128,855 |
| 合计 | 16,923,283 |

### 9.1 全部按 P1 计费

| 类型 | 页面量 | 单价 | 金额 |
|---|---:|---:|---:|
| 登录 P1 | 6,794,428 | 3.30 | 22,421.61 |
| 非登录 P1 | 10,128,855 | 2.20 | 22,283.48 |
| 合计 | 16,923,283 | - | 44,705.09 |

这与旧结算金额基本一致，可作为切换基准。

### 9.2 推荐混合场景示例

假设：

- 登录任务：P0 5%，P1 95%；
- 非登录任务：P1 30%，explore 70%。

测算金额约为：38,294.34 元。

这个场景适合 Anker 把大量探索类任务迁移到非登录 explore，整体费用低于旧方案，但紧急和生产任务仍有明确 SLA。

### 9.3 高优先级占比较高场景示例

假设：

- 登录任务：P0 10%，P1 90%；
- 非登录任务：P0 5%，P1 75%，explore 20%。

测算金额约为：44,747.23 元。

该场景与全 P1 旧口径基本持平，因为 P0 溢价被部分非登录 explore 低价抵消；若 P0 占比继续上升，应额外收取 P0 预留并发费。

## 10. 推荐落地条款

建议对 Anker 使用以下对外口径：

1. 2026-06 起账单新增 `priority`、`biz_source`、`login_required` 三个核心对账维度。
2. `P1` 价格保持原登录 3.30 / 非登录 2.20 元每千页面，保证常规任务平滑过渡。
3. 紧急任务使用 `P0`，按更高单价结算；如需 4 小时强 SLA，需购买 P0 预留并发包。
4. 当前不启用 `P2` 折扣档；如 Anker 传入 P2，结算时按 P1 单价归档。
5. 探索型需求使用 `explore + 非登录态`，低价返回公开 Top Reviews，默认不承诺全量。
6. 账号池、服务器、代理、监控告警、定制开发费用从流量费中拆出，单独列示。
7. 内部重试不重复计费；最终失败默认不计采集费；缓存复用按 20% 折算。
8. 月结单同时展示成功率、时效、回调成功率和失败原因，避免只看金额看不到服务质量。

为支持该计费方案，系统需记录以下字段：

| 字段 | 必要性 | 来源 |
|---|---|---|
| `priority` | 必须 | Anker 请求传入 |
| `biz_source` | 必须 | Anker 请求传入 |
| `login_required` | 必须 | Anker 请求传入或任务默认值 |
| `account_pool` | 必须 | 调度层路由结果 |
| `success_pages` | 必须 | 采集执行统计 |
| `records_returned` | 必须 | 结果数量 |
| `reused` | 必须 | 缓存复用标记 |

| `duration_ms` | 必须 | 任务生命周期 |
| `fail_reason` | 必须 | 失败分类 |

计费聚合 SQL/API 应支持按以下维度查询：

```text
month, workspace, biz_source, task_type, priority, login_required, account_pool
```

## 12. 建议最终结论

推荐采用“优先级 × 登录态”的阶梯单价作为主方案：

- 常规生产任务不涨价：`P1` 对齐旧单价；
- 紧急任务合理加价：`P0` 覆盖抢占和预留资源；
- 常规任务价格稳定：不存在 P2 折扣档，P2 兼容传参按 P1 计费；
- 探索任务单独低价：`explore` 降低 Anker 大规模非登录探索成本；
- 固定成本独立列示：服务器、账号池、代理、监控、定制开发不混入页面单价；
- 对账字段结构化：每月可按业务方、优先级、登录态、任务类型清楚拆分。

该方案比旧方案更适合 Anker 新需求，也能保护我们在 P0 SLA、账号池隔离和非登录探索量暴增时的成本边界。
