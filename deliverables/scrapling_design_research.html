1顶层定位

Scrapling 是一个 反爬 + 自愈解析的 Python 框架。本质：包装 patched Playwright + lxml/cssselect parser + difflib 相似度算法，做成 3 等级 fetcher。

它解决什么问题

  • 反爬天花板 — Cloudflare Turnstile / DataDome / PerimeterX 全套都能过
  • 页面改版不崩 — 把选中元素的 HTML 指纹存 SQLite，下次该 selector 找不到了能 fallback 到「相似元素」
  • API 简洁 — `StealthyFetcher.fetch(url)` 一行替代 Playwright 几十行
  • 非侵入 — 给你 Selector/Response 对象，你随便后处理
对我们的意义：smart-crawler 已经在用 StealthyFetcher（trustpilot.py / google_shopping.py / google_maps.py 三个 crawler），但只用了基础参数。Scrapling 还有一堆我们没用的反爬功能（hide_canvas / solve_cloudflare / block_webrtc / dns_over_https），后面 vidaxl_us 401 封禁可以试这些。

3三种 Fetcher 等级对比

从 Fetcher → StealthyFetcher → DynamicFetcher 是「速度 vs 反爬能力」的取舍曲线。

Fetcher底层引擎速度反爬适合什么站
Fetcher
curl_cffiHTTP/3
 curl_cffi + impersonate=chrome 极快 · 单请求 100-500ms 仅 TLS 指纹 + UA 无 JS / 无 Cloudflare 的简单 API
StealthyFetcher 🎯
patchrightcamoufox
 patchright（patched Playwright）+ 反指纹 firefox/chrome 中等 · 单页 2-5s 满级：Turnstile / WAF / canvas / webgl / webrtc 高反爬站（trustpilot/songmics/aosom/vidaxl_us）
DynamicFetcher
playwright
 原生 playwright chromium 中等 · 单页 2-5s 较弱（基础 Playwright） JS 重 + 反爬弱的站
核心 trick：StealthyFetcher 用的不是普通 Playwright，是 patchright —— 一个 patched fork（删掉了 navigator.webdriver 等可被检测的指纹）。这是为什么它能过 Turnstile 而原生 Playwright 不能的关键。

4真正的 Cloudflare 解题算法

看了源码 scrapling/engines/_browsers/_stealth.py:108，揭开它怎么解 Turnstile 的真相。

# StealthySession._cloudflare_solver(page) 真实流程

# 1. 检测 challenge 类型
challenge_type = self._detect_cloudflare(page_html)
# → "non-interactive" / "embedded" / "standard"

# 2. 等 spinner 消失（非交互型直接走这条）
while "<title>Just a moment...</title>" in page_html:
    page.wait_for_timeout(1000)

# 3. 定位 Turnstile iframe
iframe = page.frame(url=__CF_PATTERN__)
# 正则: ^https?://challenges\.cloudflare\.com/cdn-cgi/challenge-platform/.*

# 4. 拿到 iframe 的 bounding_box（绝对坐标）
outer_box = iframe.frame_element().bounding_box()

# 5. 计算点击坐标（中心 + 随机偏移 26-28px / 25-27px）
captcha_x = outer_box["x"] + randint(26, 28)
captcha_y = outer_box["y"] + randint(25, 27)

# 6. 模拟真人鼠标：100-200ms 随机按压时长
page.mouse.click(captcha_x, captcha_y, delay=randint(100, 200), button="left")

# 7. 轮询 100 次 × 100ms = 10s 等结果
while "<title>Just a moment...</title>" in page_html:
    if attempts >= 100: break
    page.wait_for_timeout(100)
关键洞察：它不是「破解」CAPTCHA，是避免被检测后正常点击通过。靠的是：
  • patchright 让浏览器看起来像真人
  • 随机化坐标 + 延迟 = 鼠标轨迹自然
  • Cloudflare 评估的是「这次访问像不像 bot」，过了打分就放行

5Adaptive Selector 自愈算法

这是 Scrapling 最 sexy 的功能。网站改版了，你的 selector 还能找到目标元素。

5.1 工作机制

# 第一次：标记元素 + 保存指纹到 SQLite
selector = page.css('.product-price', auto_save=True, adaptive=True)

# SQLiteStorageSystem 存的是元素的「unique properties」:
# - tag name (span)
# - parent tag chain (div.price-wrap > section.product)
# - 同级位置 (3rd child)
# - 关键属性 (class="price__current", data-track="*")

# 网站改版后：原 selector 找不到了，触发 relocate()
# 1. 从 SQLite 读历史指纹
# 2. 在新 HTML 树里找：
#    - 同 tag + 同 parent chain + 相同深度 → 99% 匹配
#    - 用 difflib.SequenceMatcher 比 attribute 相似度
# 3. 阈值 ≥ 0.2 即返回（默认值）

5.2 find_similar() — 横向扩散查找

# 灵感来自 AutoScraper
first = page.css('.product')[0]
all_products = first.find_similar(
    similarity_threshold=0.2,
    ignore_attributes=('href', 'src'),  # URL 类不参与对比
    match_text=False
)
# 算法：
# 1. 候选 = 同 depth + 同 tag + 同 parent tag + 同 grandparent tag
#    （3 层启发式过滤，99% 已经够准）
# 2. attribute 相似度 = SequenceMatcher(attr1, attr2).ratio()
# 3. 返回 ratio > threshold 的所有元素
为啥不撞我们的 product schema：Scrapling 只做「找到 element」这一步，element 里有什么字段、怎么映射到 SKU/Price/Title，是我们自己的业务层。它不知道也不关心。

7我们的实际调用（smart-crawler）

7.1 当前调用模板（trustpilot.py）

from scrapling.fetchers import StealthyFetcher

kw = dict(
    headless=True,
    network_idle=True,
    timeout=60000,
)
if self.proxy:
    kw["proxy"] = self.proxy

fetched = StealthyFetcher.fetch(url, **kw)
if fetched.status != 200:
    return "403/blocked"

# 解析 __NEXT_DATA__
data = self._next_data(fetched.html_content)

kw = dict(
    headless=True,
    network_idle=True,

    solve_cloudflare=True,           # 自动过 Turnstile

9从 Scrapling 偷的设计灵感

即使不深度依赖它，下面 3 点设计哲学值得抄到 smart-crawler。

① Fetcher 三等级抽象

把 fetcher 按反爬强度分 3 档（HTTP / Stealth / Browser），每档统一接口。smart-crawler 现在 17 个 crawler 各写各的，可以抽出基类 backend/app/fetchers/base.py + http.py / stealth.py / browser.py，业务 crawler 只声明用哪档。

② SQLite 指纹仓库 = 自愈基础

把抓过的 element schema 存 SQLite + 改版时 fallback，是降低 long-tail 维护成本的关键。我们 46 站手写 selector，未来肯定要走这个方向。

③ Response 对象统一

Scrapling 所有 fetcher 返同一 Response（含 status / html / json / cookies）。我们现在每个 crawler 返不同 dict，调用方都要适配。统一 Response schema 是给客户 dashboard A · Prompt-to-Table 接口的前置。

Scrapling = patched Playwright + lxml + difflib，包装成 3 档反爬 fetcher + SQLite 自愈仓库。
