0关键澄清

Scrapling 没有 user dashboard
我搜过 scrapling.dev/io/app/com/ai：要么 404，要么 scrapling.ai 是 Dynadot 待售域名（不是它的）。Scrapling 是纯 Python 库，只有 GitHub repo + readthedocs + PyPI。没有 hosted SaaS / 没有 cloud 产品 / 没有 web playground。

所以"研究它的 dashboard"实际是没东西可看。但你的真问题是：「我们能学它什么 + 怎么整合」。下面是答案。

2三层时间线 · 详细落地

① StealthyFetcher 参数升级 🟢 0.5 天

改 3 个 crawler 的 fetch 调用，从 4 参数 → 11 参数。预计破 vidaxl_us 401 + trustpilot 403。

backend/app/crawlers/ ├── trustpilot.py ← 改 StealthyFetcher.fetch 参数 ├── google_shopping.py ← 同上 ├── google_maps.py ← 同上 └── vidaxl.py ← 新增 StealthyFetcher fallback（401 时切）
# Before（4 参数）
kw = dict(headless=True, network_idle=True, timeout=60000, proxy=...)

# After（11 参数）
kw = dict(
    headless=True,
    network_idle=True,
    timeout=60000,
    proxy=...,
    solve_cloudflare=True,           # ⭐ 自动解 Turnstile
    hide_canvas=True,                # 防 canvas 指纹
    block_webrtc=True,               # 防本地 IP 泄露
    dns_over_https=True,             # 防 DNS 泄露
    real_chrome=True,                # 用本机 Chrome 而不是 chromium
    locale="en-US",                  # 配美国代理
    timezone_id="America/New_York",
)
验证方式：跑 StealthyFetcher.fetch("https://www.vidaxl.us/", **kw)，看 status 是否从 401 → 200。

② 统一 Response 对象 🟢 1 天

所有 crawler 返回同一个 CrawlResponse，给客户 dashboard "Prompt-to-Table" 接口提供统一上下文。

backend/app/ ├── fetchers/_response.py ← 新增 CrawlResponse 类 └── crawlers/ └── *.py ← 17 crawler 改返回值
# backend/app/fetchers/_response.py（新建）
from dataclasses import dataclass
from typing import Any

@dataclass
class CrawlResponse:
    site: str
    sku_count: int
    promo_count: int
    duration_sec: float
    status: str              # "success" / "blocked" / "partial"
    notes: list[str]
    verification: dict       # {sku: {url, scraped_at, screenshot, confidence}}
    raw_data: dict[str, Any]

这一步是给 v2 落地/api/agent/query + Verified Cells 接口铺路。现在 crawler 各返不同 dict，调用方都要适配，根本没法统一抽象。

③ per-site profile 持久化 cookie 🟢 0.5 天

每个高反爬站点单独维护 user_data_dir，让 Cloudflare 看到「老客户」放行。

kw["user_data_dir"] = f"./data/profiles/{site}"
# Cloudflare 评分体系会记录设备指纹 + cookie 历史
# 同一 user_data_dir 反复访问 = 老客户 = 低 challenge 率
data/profiles/ ├── vidaxl_us/ ← 自动创建，存 cookie / localStorage ├── trustpilot/ └── songmics_us/
注意：每个 profile 大约 5-20MB，46 站满量 ~1GB，NAS 没问题。但要加到 docker volumes 持久化。

④ Adaptive Selector 自愈 🟡 3-5 天

让 selector 在网站改版后自动 fallback，省 50% 长尾维护时间。

# Before（容易崩）
title = page.css('.product-title')

# After（自愈）
title = page.css(
    '.product-title',
    auto_save=True,        # 第一次成功时存指纹
    adaptive=True,         # 找不到时用指纹查相似元素
    storage=SQLiteStorage('data/scrapling_storage.db'),
)
# 网站改 .product-title → .product-name 后，自愈不崩
实施步骤：
  1. 把所有 crawler 的 css/xpath 选择器集中到 sites.yaml（部分已经做了）
  2. 包一个 SelfHealingPage wrapper，默认开 adaptive + auto_save
  3. 首次抓 46 站填充 SQLite 指纹库
  4. 加监控：当 fallback 触发率 > 10% 时告警

⑤ Fetcher 三档抽象 🟡 4-6 天

把 17 个 crawler 各写各的 fetcher 调用，抽成 3 档基类。新 crawler 只声明「我要哪档」。

backend/app/ ├── fetchers/ │ ├── base.py ← BaseFetcher 抽象（fetch / parse / dedup） │ ├── http.py ← HttpFetcher（curl_cffi · 快 · 简单站） │ ├── stealth.py ← StealthFetcher（Scrapling · 高反爬） │ ├── browser.py ← BrowserFetcher（Playwright · JS 重） │ └── __init__.py ← from_profile(site) 工厂方法 └── crawlers/ └── *.py ← 改为 fetcher = from_profile("vidaxl_us")
# 业务 crawler 大幅简化
class VidaxlCrawler:
    fetcher_tier = "stealth"  # 声明用哪档

    def crawl(self):
        page = self.fetcher.fetch(self.url)
        # 不需要重复写反爬参数

17 crawler 砍掉重复代码大约 ↓ 30%。新人加新站点只写业务解析，不碰反爬参数。

⑥ ProxyRotator 替换自研代理池 🟠 1-2 周

现有 backend/app/proxy_pool.py 是手写的（fail count / cooldown / 轮换）。Scrapling 自带 ProxyRotator 是同款。

对比项自研 proxy_pool.pyScrapling ProxyRotator
轮换✅ round-robin✅ 同款
失败计数✅ 3 次熔断✅ 内置
cooldown✅ 10min✅ 内置
per-site sticky已废弃支持 by_session
maint 工作量我们要维护0（开源维护）
不推荐立即做。现有 proxy_pool 已稳定，没出过事故。等真要做时间紧任务（如 vidaxl 全量爬）再换。

⑦ checkpoint-based Spider 框架 🟠 2 周（不推荐）

Scrapling 自带 Spider 框架，支持 Ctrl+C 暂停 / 重启续抓。看起来很 sexy 但…

不要做：
  • 我们已选 Crawlee-Python 做调度底座（W2 计划已锁）
  • Scrapling Spider + Crawlee 任务调度撞车
  • 我们的 worker.py 已经有 job queue + 重启续抓（数据库 status 字段）
Scrapling 留在 fetcher 层就好，spider 层走 Crawlee。

3明确不做的事

  • 不替换 Crawlee — Scrapling Spider 框架虽然支持 checkpoint，但调度层我们已选 Crawlee（W2 计划锁定）
  • 不把 Scrapling 升级成「底座」 — 它的定位是 fetcher，不是整个框架。我们 17 crawler 业务逻辑 + sites.yaml + 数据库 schema 都是自家的
  • 不全量切 patchright — patchright 会跟 playwright 在同一环境冲突，先在 StealthyFetcher 隔离环境里用
  • 不直接用 Scrapling 的 AI 模块 — core/ai.py 集成 LLM 抽字段，但我们已选 instructor + Pydantic + claude-haiku（参考 firecrawl 研究）

4立即可拍板的事

① 参数升级 → 立即破 vidaxl_us 401，60% 覆盖率计划解锁
② 统一 Response → 给客户 dashboard v2 接口铺路
③ per-site profile → 长效降 challenge 触发率

下下周（4-6 天）做 ⑤ Fetcher 三档抽象：17 crawler 代码 ↓ 30%
