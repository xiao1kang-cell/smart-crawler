# smart-crawler 风控策略评估报告

> 版本 v1.0｜日期 2026-05-16｜对象：遨森标杆数据采集平台 46 站 + 21 评论平台
> 评估基于：需求规格说明书 §15 站点技术档案 + 本次实地探测 + 首轮采集观测

---

## 0. 结论速览

| 维度 | 结论 |
|------|------|
| 整体可行性 | **高**。46 站中绝大多数可用「带浏览器指纹的 HTTP 直连」采集，重型浏览器仅个别站需要 |
| 最大风险点 | Vidaxl（直连 401，ASN/地域封锁）、Trustpilot（AWS WAF）、Google Shopping（SERP 反爬） |
| 代理需求 | 真正刚需住宅/静态代理的仅 ~13 站（Vidaxl 12 + Trustpilot 类）；其余直连即可 |
| 关键资产 | `static-ip-manager` 的 AT&T 静态块（60 个美国 IP）足以覆盖住宅代理需求，**自有 IP 比采购第三方住宅代理更省、更稳** |
| 预估代理月成本 | 自建静态 IP 出口方案 ≈ $0 增量（IP 已有）；若买第三方住宅代理 $500–1,500/月 |
| 法务风险 | 中。商品数据公开可采；**评论数据含个人信息，受 GDPR 约束**，需脱敏与留存策略 |

## 1. 反爬威胁全景（按品牌）

实地探测（2026-05-16）9 个标杆品牌的技术栈与反爬强度：

| 品牌 | 平台 | CDN/WAF | 直连结果 | 反爬强度 | 采集方式 |
|------|------|---------|----------|----------|----------|
| SONGMICS | Shopify | Cloudflare（未启 Bot 挑战） | ✅ 200，`/products.json` 全开放 | 低 | HTTP 直连 |
| Costway | Vue3 SPA | Cloudflare + Forter 风控 | ✅ 200，`/api/*` JSON 直连成功 | 中（**实测低于规格预期**） | HTTP 直连 API |
| Homary | Nuxt SSR | Cloudflare（图片层） | ✅ 200，SSR HTML 可解析 | 中 | HTTP 直连 |
| Yaheetech | Shopify | Cloudflare | ✅ 200，`/products.json` 开放 | 低 | HTTP 直连 |
| BEST CHOICE PRODUCTS | Shopify | Cloudflare | ✅ 200，`/products.json` 开放 | 低 | HTTP 直连 |
| VonHaus | Magento + React | — | ✅ 200 | 中 | sitemap + JSON-LD |
| Woltu | Shopware | — | ✅ 200，微数据/JSON-LD 可解析 | 低 | sitemap + 商品页解析 |
| Flexispot | React SPA | — | ✅ 200（目录小，sitemap 多为博客） | 中 | sitemap + 商品页解析 |
| **Vidaxl** | 自建平台 | **强 WAF（网段级封锁）** | ❌ **401**，首页/sitemap 均被拒 | **极高** | **必须第三方住宅代理** |

**关键发现**：
- 需求规格 §15.2 称 Costway「ASN 级封锁、必须美国住宅代理 + Playwright」。**本次从交付机器实测，curl_cffi（带 Chrome TLS 指纹）直接命中 `/api/products` 等 JSON 接口并拿到完整数据**——说明封锁是「针对国内 IP / 数据中心 ASN」的，而非全网。从美国 IP（或本机当前出口）直连即可，无需 Playwright。
- 真正硬骨头是 **Vidaxl**：12 个国家站点首页直接 401（`/waroot/` 授权页）。

**Vidaxl 封锁性质 —— 三重实测确证（2026-05-16）**：
| 测试 | 出口 IP | 手段 | 结果 |
|------|---------|------|------|
| 采集机直连 | 108.95.61.129（AT&T） | curl_cffi Chrome 指纹 | 401 |
| 经 iMac SOCKS 隧道 | 99.0.84.158（AT&T，另一 IP） | curl_cffi Chrome 指纹 | 401 |
| 真实浏览器 | 108.95.61.129 | Playwright 完整 Chromium | 401 |

> 结论：Vidaxl 封的是 **AT&T 消费网段/ASN**，不是请求指纹。换 IP（只要还在 AT&T）、
> 换浏览器都无效。**必须用与 AT&T 无关的第三方住宅代理**（Bright Data / Oxylabs 等）。
> 自有的 static-ip-manager AT&T 静态块对 Vidaxl 无效 —— 这是唯一需要花钱买代理的场景。

## 2. 评论平台风控（模块二，21 渠道）

| 平台 | 反爬强度 | 风控手段 | 对策 |
|------|----------|----------|------|
| Trustpilot（9 站） | 高 | AWS WAF + reCAPTCHA 质询 | 住宅代理 + 真实浏览器过质询；页面 JSON-LD 含完整评论 |

## 3. 分层代理策略

### 3.1 三层代理模型

| 层级 | 适用站点 | IP 类型 | 来源 |
|------|----------|---------|------|
| **L0 直连** | SONGMICS / Yaheetech / BCP / Costway / Homary / Woltu / VonHaus / Flexispot | 本机 / 服务器出口 | 无需代理 |
| **L1 静态住宅** | Vidaxl 12 站、Trustpilot、Google Map | 美国/欧洲静态 IP | **`static-ip-manager` 的 AT&T /26 块** |
| **L2 商业住宅/SERP** | Google Shopping、被 L1 封禁后的升级 | 第三方轮换住宅 | Bright Data / Oxylabs（按需采购） |

### 3.2 对接 static-ip-manager（关键，自有 IP 资产）

`static-ip-manager` 管理 AT&T 静态块 `108.95.61.128/26` —— **60 个可分配的美国静态公网 IP**
（108.95.61.129–189）。AT&T 是美国消费级 ISP，这些 IP 在反爬识别中接近「住宅 IP」，
比数据中心 IP 干净得多，且**已是自有资产，零增量成本**。

**落地方式**（推荐）：
1. 在一台持有该 ATT 线路的美国机器上，给网卡绑定多个静态 IP（`static-ip-manager` 已支持 netplan/nmcli/networksetup 分配）
2. 在该机器跑轻量代理（`3proxy` 或 `squid`），每个出口 IP 暴露一个 `http://user:pass@gw:port` 端点
3. 把这些端点逐行写入 `backend/proxies.txt` 的 `[residential]` 段
4. smart-crawler 的 `app/proxy.py` 自动轮换 —— 站点 `proxy_tier: residential` 即经此出口

```
# backend/proxies.txt 示例
[residential]
http://proxy_user:proxy_password@example-proxy-130:8080
http://proxy_user:proxy_password@example-proxy-131:8080
... （最多 60 个）
[datacenter]
http://dc-proxy-host:3128
```

> 当前 `proxies.txt` 未配置 → 所有站点走直连；Vidaxl 因此本轮会 401。
> 配好后重跑 Vidaxl 即可。代理池支持「环境变量 RESIDENTIAL_PROXY」单点 + 文件多点两种方式。

### 3.3 IP 轮换与地理匹配（规格 C-010）

- **地理匹配**：按目标站点国家选对应地区 IP（采 vidaxl.de 用德国 IP，vidaxl.com 用美国 IP）。
  AT&T 块均为美国 IP，覆盖 US 站点充分；欧洲站点需补欧洲住宅代理或接受美国 IP（多数欧洲站不强校验）。
- **轮换粒度**：建议「每站点固定一个 IP 一次采集任务」而非每请求换 IP —— 单 IP 稳定 session 比频繁换 IP 更不易触发行为风控。
- **封禁隔离**：某 IP 被某站封禁后，标记该 (IP, 站点) 组合冷却 24h，不影响该 IP 采其他站点。

## 4. 请求指纹与行为对抗

| 维度 | 现状（已实现） | 评估 | 建议 |
|------|----------------|------|------|
| TLS/JA3 指纹 | `curl_cffi` impersonate=chrome | ✅ 优秀，是直连成功的关键 | 保持；定期升级 impersonate 版本跟随 Chrome |
| User-Agent | 3 条 UA 轮换 | ⚠ 偏少 | 扩到 20+，且与 TLS 指纹版本一致（UA 写 Chrome 124 则 JA3 也要 124） |
| 请求频率 | 可配 `request_delay`=1.2s + 抖动 | ✅ 合理 | 难站调到 3–5s；按站点成功率动态调速 |
| 请求头完整性 | Accept / Accept-Language / Referer | ✅ 基本齐 | 补 `Sec-Ch-Ua`、`Sec-Fetch-*` 系列，更像真实浏览器 |
| Cookie/Session | 每站独立 Session 复用 | ✅ | 难站先访问首页拿 Cookie 再调 API（Costway 模式） |
| 并发控制 | 站点内串行 | ✅ 保守稳妥 | 跨站点可并行（已支持），站点内保持低并发 |

**结论**：当前指纹策略「curl_cffi + UA 轮换 + 限速」已足以直连 5/6 个 P1/P2 品牌。
UA 池扩充 + 补充 Sec-* 头是低成本高收益的下一步。

## 5. 验证码与 WAF 对抗

| 威胁 | 出现站点 | 当前能力 | 升级路径 |
|------|----------|----------|----------|
| Cloudflare Bot 挑战 | 多数站（未启用挑战，仅托管） | curl_cffi 可过「托管挑战」 | 启用 JS 挑战时上 `nodriver` / `botasaurus` |
| AWS WAF + reCAPTCHA | Trustpilot | 未覆盖 | Playwright + stealth 过质询；或挂打码服务（2Captcha/CapSolver） |
| 整站 WAF 拒绝（401） | Vidaxl | 未覆盖（需代理） | 住宅代理 + 完整浏览器头；仍不行则 Playwright 真实会话 |
| 行为风控（Forter） | Costway | 直连未触发 | 若后续触发：降速 + 真实鼠标轨迹（botasaurus） |

**分级响应原则**：默认 curl_cffi → 触发挑战升级 Playwright → 触发验证码挂打码服务。
不要一上来就用重型浏览器（慢 10–50 倍、易被指纹识别）。

## 6. 封禁检测与自愈（规格 C-016 / C-031 / C-032）

| 机制 | 现状 | 建议补强 |
|------|------|----------|
| 失败重试 | runner 捕获异常、任务标记 failed | 加「重试 3 次、间隔指数退避」 |
| 封禁识别 | 看 HTTP 状态 | 识别 403/401/429/验证码页/异常空结果，分类处理 |
| 数据量异常告警 | 任务表记录 products_count | 接 C-032：SKU 数骤降 >20% 自动告警（可能站点改版或被封） |
| 成功率监控 | 任务表 success_rate | 接 C-031：站点成功率 <80% 告警并自动切代理 |
| 自动降级 | 无 | 被封 → 切代理 → 降速 → 升级浏览器，逐级自愈 |

当前为 MVP 基础能力，建议把「封禁检测 + 自动切代理/降速」作为下一迭代第一优先级。

## 7. 采集节奏与错峰（规格 C-002 / C-004）

| 数据类型 | 建议频率 | 错峰时段 |
|----------|----------|----------|
| 商品价格/库存 | 每日 1 次 | 目标站当地凌晨 2–6 点 |
| 促销活动 | 每日 2 次（早/晚） | 同上 |
| 业务概览 | 每日 1 次 | 同上 |
| 评论数据 | 每周 1 次 | 周末低峰 |
| Google Shopping | 每周 1 次 | — |

调度已由 APScheduler 实现（每站点独立 cron）。错峰能显著降低被风控概率 ——
站点在低流量时段对异常流量更敏感是误区，实际相反：低峰时正常用户少，
单 IP 持续请求反而显眼，**建议错峰但配合限速 + IP 轮换，而非盲目挑凌晨**。

## 8. 法务与合规风险

| 风险项 | 等级 | 说明与对策 |
|--------|------|------------|
| robots.txt | 低-中 | 多数站 robots 仅禁用户中心/购物车，未禁商品页；采集应尊重 robots 的 Disallow 段 |
| 网站服务条款（ToS） | 中 | 部分站 ToS 禁止自动化采集；竞品监控属行业惯例但有灰度，建议法务背书、控制频率不影响对方服务 |
| 商品数据 | 低 | 公开展示的价格/标题/图片为事实性数据，可采集用于分析 |
| **评论数据（个人信息）** | **高** | 评论含评论者姓名/国家，属个人数据，**受 GDPR 约束**。对策：①只存分析所需字段 ②评论者姓名脱敏/哈希 ③不对外发布原始评论 ④设留存期限 |
| 图片版权 | 中 | 商品图有版权，仅内部分析用、不二次发布；看板缩略图建议存 URL 不落地 |
| 数据安全 | 中 | 规格 NF-03 要求 TLS 加密；采集到的竞品数据按内部机密管理 |

**建议**：评论采集模块上线前过一次法务评审，明确 GDPR 下的数据最小化与脱敏方案。

## 9. 成本估算

| 方案 | 月成本 | 说明 |
|------|--------|------|
| **自建静态 IP 出口（推荐）** | ≈ $0 增量 | 复用 static-ip-manager 的 60 个 AT&T IP + 自建 3proxy；仅一台美国机器电费 |
| 第三方住宅代理（备选/补欧洲 IP） | $500–1,500 | Bright Data/Oxylabs，按流量计费，覆盖 Vidaxl + Trustpilot + Google |
| 打码服务 | $20–80 | 2Captcha/CapSolver，仅 Trustpilot/Google 偶发验证码用 |
| SERP API（Google Shopping） | $100–300 | 若自采性价比低则采购 |

**结论**：优先榨干自有 AT&T 静态 IP 资产，第三方代理仅用于补欧洲地域 + 兜底，
可把规格预估的 $600–1,400/月压到 $100–400/月。
