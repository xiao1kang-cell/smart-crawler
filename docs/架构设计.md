# smart-crawler 服务化架构设计

> 版本 v1.0 ｜ 2026-05-16
> 目标：把当前单机脚本形态的 smart-crawler 服务化，部署到 NAS（192.168.1.80），
> 利用 NAS 大容量硬盘做长期数据存储，对外用 `smartcrawler.io` 访问。

---

## 1. 设计目标与现状

| 维度 | 现状（MVP） | 服务化目标 |
|------|-------------|------------|
| 形态 | 单机脚本 + 手动跑 CLI | 常驻服务，开机自启、故障自恢复 |
| 数据库 | SQLite 单文件 | PostgreSQL（多 worker 并发写、数据量大） |
| 采集 | 前台 / nohup 一次性 | 队列 + worker，可水平扩展、可重试 |
| 调度 | 进程内 APScheduler | 独立调度容器，把任务投递到队列 |
| 存储 | 本机磁盘 | NAS 大盘：DB + 原始快照 + 导出 + 备份 |
| 访问 | localhost:8077 | smartcrawler.io（Cloudflare Tunnel + TLS） |
| 数据留存 | 仅当前快照 | 历史快照长期归档（大盘的核心价值） |

**为什么用 NAS**：46 站全量 + 每日价格曲线追加 + 评论数据 + **原始 HTML/JSON 快照归档**，
一年体量到百万行级 + 数十～上百 GB 快照。NAS 大盘 + RAID 冗余正好承接，且数据留在内网可控。

## 3. 服务拆分（6 个容器）

| 容器 | 职责 | 副本 | 状态 |
|------|------|------|------|
| **sc-web** | FastAPI：REST API + 看板 + 鉴权 + Excel 导出 | 可多副本（无状态） | 现有代码 ✅ |
| **sc-scheduler** | APScheduler：按 `sites.yaml` cron 把采集任务写入队列 | 单例 | 现有代码 ✅ |
| **sc-worker** | 采集执行器：领队列任务 → 跑采集器 → 清洗 → 入库 → 存快照 | **可水平扩展 N** | 需新增 `app/worker.py` |
| **sc-postgres** | PostgreSQL 16，主数据库 | 单例（NAS RAID 冗余） | 需迁移（见 §6） |
| **sc-cloudflared** | Cloudflare Tunnel，smartcrawler.io 入口 | 单例 | 配置即用 |
| **sc-backup** | 定时 `pg_dump` + 快照清理（轻量 cron 容器） | 单例 | 需新增脚本 |

**「脚本化」落地**：采集逻辑已是 CLI（`python -m app.cli crawl ...`）。
worker 即一个常驻循环：`领 job → run_site() → 更新 job 状态`，复用现有 `runner.py`。
job 队列直接用现有 `crawl_jobs` 表扩展（加 `pending` 状态），**不引入 Redis**——
NAS 单机、worker 数个，DB 队列足够；并发规模上去再换 Redis（见 §8）。

## 4. NAS 存储布局（大盘的核心价值）

```
/volume1/smart-crawler/
├── pgdata/                  PostgreSQL 数据目录（容器挂载）
├── snapshots/               ★ 原始采集快照归档（大盘主要用途）
│   └── {site}/{YYYY-MM-DD}/
│       ├── products.json.gz       Shopify/Costway 原始 JSON
│       └── {sku}.html.gz          Homary/generic 原始 HTML
├── exports/                 Excel 导出归档（按日期）
├── logs/                    采集日志（按站点/日期滚动）
├── backups/                 pg_dump 每日全量备份（保留 30 天）
└── config/
    ├── sites.yaml           站点配置（热加载）
    └── proxies.txt          代理池
```

**为什么存原始快照**：①数据出错时可回溯重解析，不必重爬 ②竞品起纠纷时是证据
③采集器升级后可对历史快照重跑提取新字段 ④审计。压缩后单站单日约几 MB～几十 MB，
NAS 大盘可存数年。留存期可配（默认原始快照 180 天、导出永久）。

## 5. 数据流

**采集链路**：
`sc-scheduler`（cron 到点）→ 写 `crawl_jobs` 表一条 `pending` → `sc-worker` 轮询领取
→ 置 `running` → 调 `runner.run_site()` → 采集器抓取 → 原始响应写 `snapshots/`
→ `pipeline` 清洗去重 → 写 `sc-postgres` → `analytics` 重算趋势 → 置 `success/failed`。

**访问链路**：
浏览器 → `smartcrawler.io` → Cloudflare → Tunnel → `sc-web` → 查 `sc-postgres` → 看板渲染。

**手动触发**：看板「采集当前站点」→ `sc-web` 写一条 `pending` job → worker 领取，
与定时任务同一条队列，天然统一。

## 6. 数据库：SQLite → PostgreSQL

服务化必须换。理由：多 worker 并发写 SQLite 会锁表；数据量（已 4.4 万商品，
全量 + 历史曲线一年到百万级）SQLite 查询会慢。

**迁移成本低** —— 代码用 SQLAlchemy，仅改连接串：
```python
# db.py：环境变量驱动
DATABASE_URL = os.environ.get("DATABASE_URL",
    "sqlite:///" + str(DB_PATH))          # 本地开发仍可 SQLite
engine = create_engine(DATABASE_URL, ...)
```
- `DATABASE_URL=postgresql+psycopg://sc:pass@sc-postgres:5432/smartcrawler`
- 建表仍由 `Base.metadata.create_all` 完成
- 存量 SQLite 数据用一次性脚本导入（或直接重爬，46 站 ~4 小时）
- 依赖加 `psycopg[binary]`

WAL 那段 SQLite pragma 改为「仅 SQLite 时生效」。

## 7. 域名绑定 smartcrawler.io

用 **Cloudflare Tunnel**（NAS 不暴露任何公网端口，最安全）：

1. Cloudflare Zero Trust → Tunnels → 创建 `smart-crawler`，拿 Tunnel Token
2. Token 写入 NAS 上 `.env` 的 `TUNNEL_TOKEN`
3. `sc-cloudflared` 容器跑 tunnel
4. Public Hostname 配置：`smartcrawler.io` → `http://sc-web:8077`
5. Cloudflare 自动建 DNS CNAME + 签发 TLS 证书

→ 访问 `https://smartcrawler.io` 即看板登录页，用 `ADMIN_USERNAME` / `ADMIN_PASSWORD` 登录。

> 也可复用现有 flatkey 的 Cloudflare Tunnel 基础设施，加一条 hostname 即可。

## 8. 扩展路径（按需，不提前过度设计）

| 触发条件 | 演进动作 |
|----------|----------|
| 采集慢、要更快全量 | `sc-worker` 加副本（compose `--scale worker=4`） |
| 接入 21 个评论平台 | 新增 worker 任务类型 + 评论采集器，架构不变 |
| 并发任务量大、DB 队列吃力 | 引入 `sc-redis` + RQ/Celery 替换 DB 队列 |
| 看板访问量大 | `sc-web` 多副本 + Caddy 负载均衡；PG 加读副本 |
| 跨机房 / 异地 | worker 下沉到不同出口 IP 的机器（对接住宅代理） |

## 9. 部署清单

| 项 | 说明 |
|----|------|
| 运行平台 | NAS UGOS 自带 Docker，`docker compose up -d` |
| 编排文件 | `docker-compose.service.yml`（服务化目标编排）；`docker-compose.yml` 为单容器即用版 |
| 数据卷 | 全部挂到 `/volume1/smart-crawler/`，容器重建不丢数据 |
| 自启 | 所有容器 `restart: unless-stopped`，NAS 重启自恢复 |
| 备份 | `sc-backup` 每日 `pg_dump`；NAS 套件再叠一层卷快照 |
| 监控 | 看板「采集任务」页 + 成功率；后续可接 NAS 通知中心告警 |
| 安全 | 显式设置 `ADMIN_PASSWORD`；`SC_SECRET` 换强随机；无公网端口 |

## 10. 落地步骤（建议顺序）

1. **代码改造**（约半天）：`db.py` 支持 `DATABASE_URL`；新增 `app/worker.py` 队列消费者；
   `crawl_jobs` 加 `pending` 状态；采集器把原始响应写 `snapshots/`
2. **PostgreSQL 迁移**：起 PG 容器，建表，导入或重爬
3. **服务化 compose**：web / worker×2 / scheduler / postgres / cloudflared / backup
4. **NAS 部署**：把项目放上 NAS，建 `/volume1/smart-crawler/` 卷目录，`docker compose up -d`
5. **域名绑定**：Cloudflare Tunnel → smartcrawler.io
6. **验证**：`https://smartcrawler.io` 登录、触发采集、看快照落盘、确认每日备份

> 第 1 步是唯一需要写代码的环节，其余是部署配置。可一次性做完，也可先用
> 「单容器版（现有 docker-compose）」上线，再按本架构逐步演进。
